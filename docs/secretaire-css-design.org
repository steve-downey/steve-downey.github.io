#+TITLE: secretaire-css: Design for Palette-Aware CSS Export
#+AUTHOR: Steve Downey
#+DATE: 2026-05-10
#+OPTIONS: toc:2 num:t

* Problem Statement

The current =secretaire-css= wraps =htmlize= which resolves Emacs faces to
their computed hex colors and emits them as flat CSS rules:

#+begin_src css
.org-keyword {
    color: #b6a0ff;
    font-weight: bold;
}
#+end_src

This is opaque — there's no indication that =#b6a0ff= IS the
=magenta-alt-other= palette entry, or that =font-lock-keyword-face= maps to
it through the modus theme's face spec.  If the palette changes, you
regenerate the entire file.  If you want to tweak one color, you have to find
which face uses it by hex value.

A curated CSS file would look like:

#+begin_src css
:root {
    /* modus-vivendi-tinted palette */
    --bg-main: #0d0e1c;
    --fg-main: #ffffff;
    --magenta-alt-other: #b6a0ff;
    --green: #44bc44;
    --blue-alt: #79a8ff;
    --cyan: #00d3d0;
    --red: #ff8059;
    --yellow: #d0bc00;
    /* ... only palette entries actually used by faces we export */
}

/* font-lock-keyword-face → magenta-alt-other */
.org-keyword {
    color: var(--magenta-alt-other);
    font-weight: bold;
}

/* font-lock-string-face → green */
.org-string {
    color: var(--green);
}
#+end_src

This is what "working with the grain" means: the CSS parallels the theme's
semantic structure.  A human reading it can see the design intent.  Changing
the palette is a `:root` edit.  The face→color mapping is documented inline.

* Architecture: How Modus Themes Work

** Palette

A theme's palette is an alist: =(semantic-name . hex-color)=.

#+begin_src emacs-lisp
;; modus-themes 3.x (Emacs 29):
modus-themes-vivendi-colors
;; => ((bg-main . "#000000") (fg-main . "#ffffff") (red . "#ff8059") ...)

;; modus-themes 4.x (Emacs 30+):
(modus-themes-get-color-value 'blue)
;; => "#2fafff"
#+end_src

Palette entries are semantic: =bg-main=, =fg-dim=, =red=, =blue-alt-other=,
=magenta-faint=.  Some are structural (=bg-main=, =fg-main=) and some are
color roles (=red=, =green=, =cyan=).

** Face Specs

Faces reference palette names:

#+begin_src emacs-lisp
`(font-lock-keyword-face ((,class :inherit modus-themes-bold
                                  ,@(modus-themes--syntax-extra
                                     magenta-alt-other magenta-alt-other-faint
                                     cyan cyan-faint))))
#+end_src

This means: under the default syntax style, keyword face uses
=magenta-alt-other=; under the =faint= style, it uses =magenta-alt-other-faint=;
under =alt-syntax= it uses =cyan= / =cyan-faint=.

** Theme Loading

When a theme loads, it evaluates face specs with the palette bound, producing
resolved face attributes (hex colors).  =htmlize= then reads these resolved
attributes.

* Design: secretaire-css v2

** Goals

1. Export CSS that uses CSS custom properties for palette colors
2. Include only palette entries that are actually used by exported faces
3. Annotate each face rule with the Emacs face name and palette color name
4. Produce output that looks hand-curated
5. Work with any modus variant (vivendi, operandi, tinted, deuteranopia, tritanopia)
6. Work with the theme's own API rather than scraping resolved face attributes

** Strategy

Instead of =htmlize='s approach (render text, extract colors from
properties), we:

1. Load the target theme
2. Get the palette: =(modus-themes-current-palette)= or equivalent
3. Get the face specs: inspect =modus-themes-faces= or query each face's
   attributes and reverse-map hex → palette name
4. For each face we care about (font-lock-*, org-*, etc.):
   a. Get the resolved foreground/background/weight/slant
   b. Reverse-lookup hex → palette name
   c. Emit a CSS rule using =var(--palette-name)=
5. Emit a =:root= block with only the palette entries we referenced

** The Reverse-Mapping Problem

The hard part: given a face's resolved =#ff8059= foreground, map it back to
=red= in the palette.  This is straightforward — it's just an alist lookup
reversed:

#+begin_src emacs-lisp
(defun secretaire-css--reverse-palette (palette)
  "Create a hash table mapping hex values to palette names."
  (let ((table (make-hash-table :test #'equal)))
    (dolist (entry palette)
      (when (stringp (cdr entry))
        (puthash (downcase (cdr entry)) (car entry) table)))
    table))
#+end_src

Note: multiple palette entries might share the same hex value (unlikely in
modus but possible in theory).  Use the first match.

** Non-Modus Themes

The package assumes a modus-pattern theme is loaded.  There is no htmlize
fallback.  The =htmlize= dependency has been removed.  If a theme without a
palette API is loaded, =secretaire-css-generate= signals an error.

** CSS Output Structure

#+begin_src css
/* secretaire-css: modus-vivendi-tinted
 * Date: 2026-05-10 14:30:00
 *
 * Palette entries referenced by exported faces.
 * Source: (modus-themes-current-palette)
 */
:root {
    --bg-main: #0d0e1c;
    --fg-main: #ffffff;
    --fg-dim: #e0e6f0;
    --red: #ff8059;
    --green: #44bc44;
    --blue-alt: #79a8ff;
    --cyan: #00d3d0;
    --magenta-alt-other: #b6a0ff;
    --yellow: #d0bc00;
    /* ... */
}

/* === Core font-lock faces === */

/* font-lock-keyword-face: magenta-alt-other (bold) */
.org-keyword {
    color: var(--magenta-alt-other);
    font-weight: bold;
}

/* font-lock-string-face: green */
.org-string {
    color: var(--green);
}

/* font-lock-comment-face: fg-dim (italic) */
.org-comment {
    color: var(--fg-dim);
    font-style: italic;
}

/* font-lock-type-face: cyan */
.org-type {
    color: var(--cyan);
}

/* === Org-mode faces === */

/* org-code: green */
.org-org-code {
    color: var(--green);
}

/* ... */
#+end_src

** Face Selection

Not all 1000+ faces need to be exported.  Export:
1. All =font-lock-*= faces (core syntax highlighting)
2. All =org-*= faces (org-mode structure)
3. Faces from loaded major modes that font-lock references
4. The =default= face (for =-bg-main= / =-fg-main=)

A configurable allowlist controls which face prefixes to include.

** Implementation Phases

*** Phase 1: Palette extraction (done in current code as =--palette-vars=)
Extract bg/fg from default face → emit as CSS vars.

*** Phase 2: Reverse-map + annotated export
For each exported face:
- Resolve current attributes
- Reverse-map hex to palette name
- Emit CSS rule with =var(--name)= and comment

*** Phase 3: Deduplication
Multiple faces may share the same rendered CSS.  Group them:
#+begin_src css
/* font-lock-doc-face, font-lock-doc-markup-face: green-faint (italic) */
.org-doc,
.org-doc-markup {
    color: var(--green-faint);
    font-style: italic;
}
#+end_src

*** Phase 4: Minimal palette
Only emit =:root= entries for palette names actually referenced in rules.
Track usage during generation.

* Compatibility

** modus-themes 3.x (Emacs 29)

- Palette: =modus-themes-vivendi-colors= / =modus-themes-operandi-colors=
- Access: =(modus-themes-color 'red)= → ="#ff8059"=
- Macro: =modus-themes-with-colors= binds all palette names as variables

** modus-themes 4.x (Emacs 30+)

- Palette: =(modus-themes--current-theme-palette)= or similar
- Access: =(modus-themes-get-color-value 'red)=
- Tinted variants: =modus-vivendi-tinted=, =modus-operandi-tinted=, etc.
- Each variant has its own palette (tinted uses warmer background)

** Non-modus themes

Fall back to flat hex export (current htmlize approach).

* Relationship to htmlize

The v1 approach wrapped =htmlize-region= which rendered text to HTML and
extracted the =<style>= block.  The v2 approach bypasses htmlize entirely:

1. Queries each face's resolved attributes via =(face-attribute face :foreground nil t)=
2. Reverse-maps hex → palette name via the modus palette alist
3. Emits CSS with =var(--palette-name)= references directly

The =htmlize= package is no longer a dependency.

* DRY Implications

With the palette in =:root=:
- =tailwind-extras.css= uses =var(--bg-main)= instead of =#0d0e1c=
- No color value appears in more than one place across all CSS files
- Adding a new modus variant means regenerating the file; the structure stays the same
- The palette section is the single source of truth for all syntax colors

* Open Questions

1. Should we export font-weight/font-style as CSS custom properties too?
   (Probably not — those are face attributes, not palette values)

2. Should the exported file include faces that inherit and add nothing?
   e.g., =font-lock-comment-delimiter-face= inherits =font-lock-comment-face=.
   Emit as CSS inheritance: =.org-comment-delimiter { /* inherits .org-comment */ }=?
   Or just omit?

3. How to handle the =org-html-htmlize-font-prefix= ("org-") mapping?
   Face =font-lock-keyword-face= → class =.org-keyword= (strip "font-lock-" and "-face").
   This mapping is done by htmlize.  We need to replicate it.

4. For the 4.x "semantic mappings" layer (e.g., =keyword= maps to =magenta-alt-other=
   through the mappings alist), should we show both levels?
   e.g., =/* keyword → magenta-alt-other → #b6a0ff */=

* Next Steps

This design doc captures the direction.  Implementation requires:
- Access to modus-themes 4.x (or 3.x with tinted variants from MELPA)
- Testing with the actual face→palette reverse mapping
- Deciding on the face allowlist
- Building the CSS emitter

The current htmlize-based export continues to work as a baseline.  The
palette-aware export is an opt-in upgrade activated when modus-themes is
detected.
